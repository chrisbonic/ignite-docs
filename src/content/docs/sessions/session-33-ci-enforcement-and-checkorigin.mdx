## What shipped

### 1. CI enforcement across all 4 repos

GitHub Actions workflows updated to run `format:check` and (where applicable) `typecheck` before `build`. All steps use `if: always()` so every check runs on every push, surfacing all failures instead of stopping at the first.

- **compute** (parent CI workflow): `format:check` → `build` (tsc --noEmit). `format:check` is new.
- **admin**: `format:check` → `typecheck` → `build`. Both new.
- **site**: `format:check` → `typecheck` → `build`. Both new.
- **docs-site**: `format:check` → `typecheck` → `build`. Both new.

Philosophy stays advisory per low-stakes platform rules — checks report failure but don't gate merges beyond what `build` already did.

### 2. `@types/node` explicitly declared in admin + site

First CI run exposed that admin and site were implicitly resolving `@types/node` through hoisted `node_modules`. CI's clean `npm ci` environment had no such hoisting — every file reading `process.env` (middleware, auth, config, agent workers) hit `Cannot find name 'process'`. Admin CI reported 70 errors, site reported 6.

Fix: `npm install --save-dev @types/node@^22.0.0` in both repos. Local typecheck now matches CI — green for the right reason, not by accident.

### 3. Astro `security.checkOrigin: true` disabled in admin + site

**This was not planned work.** Mid-session, production admin saves began returning 403 "Cross-site POST form submissions are forbidden" on every page and partner edit. This was a genuine production incident triggered by Webflow infrastructure changes during the session.

**Root cause:** Webflow Cloud forwards requests to the underlying Cloudflare Worker under an **internal hostname** of the form `<worker-id>.wf-app-prod.cosmic.webflow.services`. Inside the Worker, `context.url.origin` resolves to that internal host. The browser's `Origin` header is the public domain (`https://www.ignite-studio.co`). Astro's built-in `checkOrigin` middleware compares them — `isSameOrigin = request.headers.get("origin") === url.origin` — and they can never be equal. Every POST gets 403.

Confirmed via temporary debug response headers:

- `x-debug-origin`: `https://www.ignite-studio.co`
- `x-debug-url-origin`: `https://6293f40d-6a30-47c4-a757-fab052c7af83.wf-app-prod.cosmic.webflow.services`

**Fix:** `security: { checkOrigin: false }` in both `admin/astro.config.mjs` and `site/astro.config.mjs`. Admin uses HMAC session cookies for auth, so CSRF risk is bounded by an already-authenticated session being tricked. Acceptable for a single-user invite-only admin.

### 4. Incident writeup in `.claude/rules/webflow-cloud.md`

Added two sections:

- **Astro version postscript** — Webflow silently re-enabled Astro 6 support, then reverted it again on 2026-04-16, without customer notification. Confirmed via support ticket #1116991.
- **`checkOrigin` must be `false`** — full architectural explanation, why it's a platform issue not a version issue, and a "do not re-enable" directive that survives future Astro version changes.

Source material for the Webflow CTO meeting next week.

## How to apply

- **CI now enforces formatting + types on every push.** Don't let new format violations or type errors land. The advisory posture means they surface but don't block — fix them in the next commit, not via a bypass.
- **`checkOrigin` stays `false`** on admin and site regardless of Astro version, until Webflow changes how they forward the Host header to Workers. This is an architectural constraint, not a version compatibility one.
- **Webflow Cloud is actively thrashing platform versions without notice.** Assume the floor can shift under you. Don't ship code that depends on niche Astro middleware behavior without verifying it survives a cold deploy.
- **Local typecheck can lie about CI** — if a type check passes locally for the first time after adding it to CI, run it in a clean container before trusting it. Hoisted `@types/*` packages don't replicate in `npm ci`.
