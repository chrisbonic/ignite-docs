## What shipped

### 1. `createdBy` threaded end-to-end on KV jobs

The "light tracking" follow-through promised in Session 34. Every job record in KV is now stamped with `{ username, displayName, role }` of the operator who initiated it. Two job stores, two flows.

**Type surface (new field on both interfaces):**

- `AgentJob.createdBy` — [admin/src/lib/agent-jobs.ts](../../../../../admin/src/lib/agent-jobs.ts), [compute/lib/agent-jobs.ts](../../../../../compute/lib/agent-jobs.ts)
- `GenerateJob.createdBy` — [compute/lib/generate-jobs.ts](../../../../../compute/lib/generate-jobs.ts)

Shape:

```ts
createdBy?: { username: string; displayName: string; role: 'admin' | 'sales' };
```

Optional — legacy pre-35 jobs have no `createdBy` and render as `-` / `Unattributed` in rollups.

**Write path (admin side):**

- [admin/src/lib/agent-jobs.ts](../../../../../admin/src/lib/agent-jobs.ts) — `createJob` accepts `createdBy` arg, stamps onto job
- [admin/src/pages/api/agents/prepare.ts](../../../../../admin/src/pages/api/agents/prepare.ts) — reads `Astro.locals.operator`, derives `createdBy`
- [admin/src/pages/api/agent-run.ts](../../../../../admin/src/pages/api/agent-run.ts) — Agent Mode SSE proxy forwards `createdBy` in compute prepare body

**Write path (compute side):**

- [compute/api/agent/prepare.ts](../../../../../compute/api/agent/prepare.ts) — accepts `createdBy` in body, passes to `createJob`
- [compute/api/generate/start.ts](../../../../../compute/api/generate/start.ts) — accepts `createdBy` in body, passes to `createGenerateJob`

**Browser → compute (direct calls bypass admin server):**

The Pitch Agent (`agents/pitch.astro`) and Agent Mode (`agent.astro`) browser code calls compute directly. For these, the operator is exposed to the client via a JSON script tag in [admin/src/layouts/AdminLayout.astro](../../../../../admin/src/layouts/AdminLayout.astro):

```astro
<script is:inline type="application/json" id="operator-data"
  set:html={JSON.stringify(operator ? {...} : null)} />
<script is:inline>
  window.currentOperator = JSON.parse(...);
</script>
```

All browser-initiated POSTs include `createdBy: window.currentOperator || undefined`:

- [admin/src/pages/agent.astro](../../../../../admin/src/pages/agent.astro) (Agent Mode prepare)
- [admin/src/pages/agents/pitch.astro](../../../../../admin/src/pages/agents/pitch.astro) (Pitch Agent prepare + retry)
- [admin/src/pages/clients/[id]/generate.astro](../../../../../admin/src/pages/clients/[id]/generate.astro)

### 3. Logout middleware fix

**Root cause (surfaced during prod testing):** `/api/auth/logout` was not in `PUBLIC_PATHS` in [admin/src/middleware.ts](../../../../../admin/src/middleware.ts). When a user clicked Sign Out, the middleware checked for a valid session, returned `401 {"error":"Unauthorized"}` JSON, and never let the route handler run to delete the cookie and redirect to `/login`. Added `/api/auth/logout` to `PUBLIC_PATHS`.

Logout now: deletes cookie, redirects to `/login`, cleanly.

## How to apply

- **When adding new generation workers or agent flows:** add `createdBy` to the request body from the browser (via `window.currentOperator`) and from any server-side forwarder (via `Astro.locals.operator`). Pass it into `createJob` / `createGenerateJob` as the last arg.
- **When adding new admin API routes that mutate state:** remember `Astro.locals.operator` is available on every non-public route. Use it.
- **When adding public (pre-auth) API routes:** add the path to `PUBLIC_PATHS` in [admin/src/middleware.ts](../../../../../admin/src/middleware.ts). Logout was the canary for this pattern. Watch for similar gotchas on any future unauthenticated-but-authoritative route.
