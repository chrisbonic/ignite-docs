## 2026-04-11 — Astro 6.x rejection

**Impact:** 6 consecutive admin deploys failed (commits from `4dbb525` through `91e007b`). Production admin frozen at `a4618c5` (Premium Note template, April 10).

**Error:**

> Astro 6.x is not yet supported by Webflow Cloud. Please use Astro 5.x. You can downgrade by running: npm install astro@^5.0.0

**Root cause:** Webflow Cloud's builder added an Astro 6.x guard. Cloudflare acquired Astro on 2026-01-16 and Astro 6 entered beta around the same time. Webflow's adapter/runtime is not yet validated against Astro 6.x.

**Not a code change on our side** — the `admin/` `package.json` has had `"astro": "^6.0.5"` since the initial commit. Successful deploys through April 10 happened because the guard did not exist yet.

**Status (as of 2026-04-15):** Chris has an open support ticket with Webflow. **Not downgrading** until we hear back — a 6→5 regression is significant work (content layer, session API, vite plugin shifts) and should only happen if Webflow confirms no imminent 6.x support.

**Workaround:** None. Unshipped work on `main`:

- Session 27 outreach UX improvements
- Session 28 security hardening (CSRF, rate limiting, error sanitization)
- Session 29 performance optimization (parallel CMS calls, TTL caching, schema doc)

**Trace ID (admin deploy `91e007b`):** `388785662812514010`

**Context links:**

- [Cloudflare acquires Astro (Jan 16, 2026)](https://blog.cloudflare.com/astro-joins-cloudflare/)
- [Webflow Cloud framework customization docs](https://developers.webflow.com/webflow-cloud/environment/framework-customization)

## 2026-04-15 — Resolution: Astro 6 → 5.18 downgrade + React deps re-added

**Impact:** Sessions 27/28/29 work unblocked. Admin production moved from `a4618c5` (frozen April 10) to `9937ea8`. Site production moved from `8d86c79` to `ac803c0`.

**Trigger:** Webflow support confirmed on 2026-04-15 that Astro 6.x is not supported and has no ETA.

**Change 1 — Astro downgrade (both repos):**

- `admin/package.json`: `astro: ^6.0.5` → `^5.18.0` (commit `80eacdf`)
- `site/package.json`: `astro: ^6.0.5` → `^5.18.0` (commit `ac803c0`)
- `@astrojs/cloudflare@^12.6.13` works with both majors — no adapter change needed.
- `@astrojs/react@^5.0.3` resolved fine under Astro 5 in site/ — no integration change needed.
- **Zero code changes.** Our API surface (routes, middleware, Cloudflare adapter, `security.checkOrigin`) is stable across Astro 5 and 6. Local `npm run build` passed first try in both repos.

**Change 2 — React deps re-added to admin (commit `9937ea8`):**

First admin deploy attempt on Astro 5 failed with:

> `Cannot find package 'react-dom' imported from /repo/node_modules/@astrojs/react/dist/version.js`

**Root cause:** Webflow Cloud's builder (v1.2.0) now auto-installs `@astrojs/react@^4.0.0` as a framework requirement ("Required for Webflow React components"). Admin had removed `react` and `react-dom` during Session 27 cleanup (commit `ac438e8`, 2026-04-10) because admin uses no React components. The builder's auto-installed React integration then failed to resolve the peer.

**Fix:** Added `react: ^19.2.4` and `react-dom: ^19.2.4` back to `admin/package.json`. Site already had these. Admin deploy succeeded after push.

**Lessons captured:**

1. Even apps with no React components must keep `react` + `react-dom` in dependencies on Webflow Cloud. Documented in [.claude/rules/webflow-cloud.md](../../../.claude/rules/webflow-cloud.md) under "React deps — required even for non-React apps."
2. "Your package.json works on their builder today" is not a durable contract. Their builder can change; ours is tested by the same checks that ship.
3. The 6→5 downgrade being trivial was a genuine surprise — we had budgeted for content-layer, session-API, and vite-plugin regressions. None materialized because we don't use those APIs.

**Unblocked work that now ships:**

- Session 27 outreach UX improvements
- Session 28 security hardening (CSRF, rate limiting, error sanitization)
- Session 29 performance optimization (parallel CMS calls, TTL caching, schema doc)

**Sales impact:** 4 business days of production freeze during active prospect demos. Sessions had to either avoid touching production-facing screens or demo from local admin.

## 2026-04-16 — Astro 6 silent enable/revert + `checkOrigin` platform incompatibility

**Impact:** Every admin POST endpoint returned `403 Cross-site POST form submissions are forbidden` in production. All page edits, partner edits, and outreach saves broken. Dozens of failed save attempts from the primary operator. Discovered during active sales work.

**Observed behavior:** Forms rendered fine. Clicking Save returned 403 with an Astro-generated error body. Browser Network panel showed the POST hitting `/admin/api/.../update` with a matching `Origin: https://www.ignite-studio.co` header and `:authority: www.ignite-studio.co`. Same-origin from the browser's perspective, yet Astro rejected it as cross-site.

**Root cause (confirmed via instrumentation):**

Webflow Cloud forwards requests to the underlying Cloudflare Worker under an **internal hostname** of the form `<worker-id>.wf-app-prod.cosmic.webflow.services` — not the public domain. Inside the Worker, `context.url.origin` resolves to that internal hostname. Astro's built-in `checkOrigin` middleware does a strict string comparison:

```js
// astro/dist/core/app/middlewares.js
const isSameOrigin = request.headers.get('origin') === url.origin;
```

Confirmed by emitting the two values as debug response headers on a temporary deploy:

- `x-debug-origin`: `https://www.ignite-studio.co`
- `x-debug-url-origin`: `https://6293f40d-6a30-47c4-a757-fab052c7af83.wf-app-prod.cosmic.webflow.services`

The two strings can never be equal on Webflow Cloud, so every POST returns 403 when `checkOrigin: true`. This is an architectural incompatibility, not a configuration problem.

**Webflow-side trigger:** The incident appeared today, not earlier. Ticket #1116991 with Webflow Customer Support confirmed that Webflow **silently re-enabled Astro 6 support** between 2026-04-15 and 2026-04-16 (after telling us on 2026-04-15 that there was no ETA), then **reverted it again** without customer notification. Paul from Webflow Support: "As a follow-up, we had to revert our support for Astro 6, so please hold off on upgrading to Astro 6 until later when we've re-introduced support for Astro 6 again."

We were not on Astro 6 — we were pinned to 5.18 since Session 31. But during their enable/revert window, their routing behavior shifted such that `context.url.origin` inside the Worker started reflecting the internal hostname. Before this window, admin had run with `checkOrigin: true` and all POSTs worked. After: 403 on every POST.

**Fix:**

1. Temporary: set `checkOrigin: false` in [admin/astro.config.mjs](https://github.com/chrisbonic/ignite-webflow-pitch-admin/blob/main/astro.config.mjs) (commit `018c205`, saves immediately restored).
2. Instrumented middleware with debug response headers to capture the actual `url.origin` value (commit `018c205`).
3. Confirmed root cause from the debug headers.
4. Removed debug logging, kept `checkOrigin: false`, documented the architectural reason in source comments (commit `f511a21`).
5. Applied `checkOrigin: false` preemptively to [site/astro.config.mjs](https://github.com/chrisbonic/ignite-webflow-pitch-site/blob/main/astro.config.mjs) (commit `eed320a`).
6. Full incident captured in [.claude/rules/webflow-cloud.md](../../../.claude/rules/webflow-cloud.md) with a permanent "do not re-enable" directive.

**Security trade-off:** Admin uses HMAC session cookies for auth, so residual CSRF risk is bounded by an already-authenticated session being tricked into submitting a crafted form elsewhere. Acceptable for a single-user invite-only admin.

**Why the check can never work on Webflow Cloud:**

The mismatch is not an Astro bug. Astro's same-origin check is correct — the inbound URL should equal the outbound Origin. But on Webflow Cloud, the inbound URL is rewritten to an internal hostname before it reaches our Worker, while the browser's Origin header remains the public domain. No amount of Astro configuration can bridge this. Fixing this would require either Webflow preserving the public Host header when forwarding to Workers, OR our code intercepting before Astro's middleware and synthesizing a matching URL (fragile and version-coupled).

**Lessons captured:**

1. **"Astro downgrade is behaviorally neutral" was an over-claim.** Session 31 asserted the 6→5 downgrade had no behavioral impact because our API surface (routes + middleware + Cloudflare adapter) was stable across both majors. We did not audit Astro's internal middleware behavior. When making such claims, scope them narrowly.
2. **Webflow Cloud is actively thrashing platform versions without customer notification.** Between an official "no ETA" response and 24 hours later, they rolled Astro 6 out and back. Do not depend on platform behavior being stable across their silent rollouts.
3. **Diagnose before prescribing.** First instinct during the 403 was to propose three plausible-sounding fixes. User (correctly) forced a pause to gather actual request/response data before flipping config. The real cause was not on the initial guess list.
4. **`checkOrigin: true` is permanently incompatible with Webflow Cloud's proxy architecture** until they change how they forward the Host header. Stay off it regardless of Astro version.

**Sales impact:** 1+ hour of operator-visible production breakage during active sales prep. Operator was actively trying to edit outreach pages for a prospect demo. Incident resolved with saves restored within that window.
